# retrospective-codify

## 目的

セッション中に発生した手戻り・試行錯誤・ユーザーからの訂正を、次回以降再発しない形に固定する。一次成果物（コード）と切り離し、**運用知識**としてリポジトリまたはグローバル設定に書き戻すためのスキル。

## いつ使うか

- ユーザーが `/retrospective` と入力したとき
- 長めのタスクが完了したあとに、再利用可能な気付きが複数残ったとき
- 同じ指摘・誤りを 2 回以上したとき（即座に固定する）

## 固定先の判断

抽出した知見ごとに、次の 4 箇所のいずれに置くかを決める。

| 種類 | 固定先 | 例 |
|---|---|---|
| 手順・プレイブック | skill（`claude/skills/<name>/SKILL.md` か repo 内の `.claude/skills/`） | 「CI 失敗時はまず〜を確認」 |
| ふるまい方針 | CLAUDE.md（global または project） | 「テストは正常系/異常系を別関数に」 |
| 自動検査可能なルール | hook（`settings.json` の hooks） | 「コミット前に nixfmt を実行」 |
| 静的に書ける規約 | linter / formatter 設定 | `stylua.toml`, `.editorconfig` 等 |

判断基準:

- **skill にするのは**: 複数ステップがあり、状況に応じた分岐があるもの
- **CLAUDE.md に書くのは**: 毎回守ってほしい短い指針
- **hook にするのは**: 人が見張らなくても機械的に強制できるもの（静的検査可能）
- **linter/formatter にするのは**: そのツールで表現できるもの（プロンプトに書かない）

判断不能なときはユーザーに確認する。

## 配置先（project 固有 vs グローバル）

- **project 固有**: 特定 repo のドメイン知識・規約・ファイルレイアウトに依存 → `<repo>/.claude/skills/` や `<repo>/CLAUDE.md`
- **グローバル**: 言語・ツール横断で再利用可能 → この dotfiles の `claude/skills/` や `claude/CLAUDE.md`
- **迷ったら聞く**。後から移動するとパス参照が壊れやすい。

## 手順

1. **抽出**: セッション中の失敗⇄成功の対応付けを箇条書きで列挙する。ユーザーからの訂正・「そうじゃない」発言は特に重視する。
2. **一般化**: プロジェクト固有名・秘密値・具体パスを除いて再利用可能な形にする。固有名は `<REDACTED>` や `<PROJECT_ROOT>` に置換。
3. **重複チェック**: 既存の `claude/CLAUDE.md` / `claude/skills/*` / `claude/commands/*` / repo の CLAUDE.md を grep し、同趣旨のルールが既にないか確認する。あれば更新案を出す。
4. **提案**: 以下のフォーマットでユーザーに提示する。**このターンでは書き込まない**。

   ```
   ## 気付き 1: <短い見出し>
   - 一般化した内容: ...
   - 固定先: skill / CLAUDE.md / hook / linter のどれか、理由も
   - 配置先: project 固有 / グローバル、理由も
   - 具体的な差分（diff 案）:
   ```

5. **採用指示を待つ**。`保存` / `skill 2 だけ採用` など明示の指示があってから Edit/Write を行う。
6. **書き出し**: 承認された項目のみ反映し、作成・変更したパスを箇条書きで出力する。

## アンチパターン

- 承認なしに書き込む
- 固有名詞をそのまま残す
- 「気をつける」系の抽象的な文言で固定する（検証可能な表現に落とす）
- 同じ内容を CLAUDE.md と skill の両方に二重化する
- hook で書けるものを自然言語でルール化する
